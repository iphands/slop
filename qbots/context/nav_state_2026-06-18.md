## What WORKS now (committed)

- **`usercmd.msec` derived from the real frame dt** (was hardcoded 33 → bots ran at 1/3

- **Pure-pursuit steering** (`nav.rs`): project the bot onto the path polyline, aim a FIXED
  `LOOKAHEAD=96u` ahead from the projection. Density-independent; gave grid=24 stable 24/24

- **Corner-escape recovery** (`scenario.rs` BackOffThenRepath): when hard-stuck, steer

## The KEY strategic finding (the RL)

The rocket launcher (`704,104,912`) sits on a z=920 platform. We thought the only route was
a narrow "drop onto a ledge" path the user described. The real situation, proven with
`navinspect path`:

- **grid=24:** spawn5→RL = 23 hops, the path must DETOUR down off z=920 (the platform is
  poorly connected at 24u sampling). The RL is reachable but the path is long → ~6/24 reach.
- **grid=12:** spawn5→RL = 44 hops but stays ENTIRELY on z=920 — a DIRECT platform route.
  Finer sampling connects the platform. So **fine grid OPENS the RL** (no narrow-ledge
  modelling needed).

BUT the weapon test is WORSE at grid=12 (3/24 vs 6/24) because **grid=12 navigation is worse
(~14/24 vs 24/24)**. The connectivity win is cancelled by the nav-quality loss.

> **Conclusion: the RL is gated on grid=12 nav reliability.** Solve that and the RL follows.
> (We also tried adaptive narrow-ledge sub-sampling in `generate()` — it added nodes but
> hurt nav, the same density problem; reverted.)

## The UNSOLVED problem: fine-grid navigation reliability

At grid=12 the nav graph is denser (4× nodes). spawn-to-spawn ≈ 14/24 (vs 24/24), with high
variance (10-18). With count=10 (minimal crowding) it's 4-8/10 vs grid=24's 10/10 — so it's
**nav quality, not bot crowding**. Dominant failure: bots get HINDERED ~20s pressing
corner / wall-adjacent nodes; give-up/recovery churn but can't escape.

### What's been tried (the important lesson)
- **ARCHITECTURAL changes WORKED:** pure-pursuit steering (grid=24 stable), corner-escape
  recovery (grid=12 11→14).
- **EVERY heuristic/parameter TWEAK HURT or was neutral:** projection-sync advance,
  forward-node fallback, progress-based give-up reset (dropped grid=24 to 16!),
  wall-avoidance steering, finer grid alone, grid 16/18/20/22 sweeps, density radius tuning.

**Why tweaks backfire:** they ADD a second progress signal on top of the existing
reach/orbit/give-up logic in `NavigationDriver::update()`. Two progress systems disagree at
high density (32u/tick over 12u nodes), so the bot either churns (gives up on passed nodes)
or never gives up (grinds a bad path). Surgical add-ons can't win.

### Grid-coupling facts learned (for whoever does the rewrite)
- The load-bearing quantity is the **connection RADIUS** (`CONNECT_RADIUS=72u`), not the
  grid. At fine grids a fixed 72u radius = dense mesh (90 edges/node) → capped to
  `MAX_CONNECT_CELLS=3`. Non-divisor grids (16, 20) can't represent 72u exactly → awkward;
  divisor grids (24, 18, 12, 9, 8) are clean.
- Steering constants currently derive from `world::GRID_SPACING` (`WP_REACH_HORIZ`,
  `WP_REACH_DZ`, `ORBIT_RADIUS`). `LOOKAHEAD` is a FIXED 96u (pure-pursuit is density-indep).

## THE NEXT STEP (well-defined; do it as its own plan)

**Rewrite `NavigationDriver::update()` progress logic to be projection-native.** Replace the
reach/orbit/give-up trio with a SINGLE model driven by the bot's arc-length projection on
the path polyline:

1. **Progress** = arc-length of the bot's projection onto `current_path` (helper already
   sketched twice: `project_onto_path` + a `path_progress` accumulator).
2. **Advance** `current_waypoint` to the node just ahead of the projection.
3. **Give-up** fires only when the projection STALLS (no arc-length gain) for
   `GOAL_GIVEUP_TICKS` — NOT when a reach radius is missed.
4. **Reached (goal)** = projection within the last segment AND close to the final node.
5. **Preserve** the ledge / false-edge handling (the `orbit_force` + dz>LEDGE_DZ branch that
   blacklists false bridges) — grid=24's 24/24 depends on it. Trigger it on stall-near-a-
   large-dz-node instead of orbit.

**Constraints / guardrails:**
- grid=24 spawn-to-spawn must stay 24/24 (hard regression gate every step).
- It is a REPLACEMENT, not an addition — remove `WP_REACH_*`/`ORBIT_*`/`near_wp_ticks`
  reach logic, don't layer on top (layering is what failed).
- Validate at grid=12 (target: match grid=24), then re-check the RL weapon test (should
  jump once grid=12 nav is reliable, because connectivity is already better there).

**Verification recipe:**
```
qbots generate-map-cache --map q2dm1 --spacing 24   # ~3s
qbots generate-map-cache --map q2dm1 --spacing 12   # ~34s
qbots spawn-to-spawn  --count 24 --max-secs 60 --spacing 24   # must stay 24/24
qbots spawn-to-spawn  --count 24 --max-secs 60 --spacing 12   # target: ~24/24
qbots spawn-to-weapon rocketlauncher --count 24 --max-secs 60 --spacing 12  # RL payoff
```
