# 3ZB2 Pathing & Navigation Analysis

**Source:** `vendor/3zb2-zigflag/src/`  
**Date Analyzed:** 2026-06-16  
**Version:** 3ZB2 098 (1999)

---

## TL;DR

**3ZB2 does NOT parse BSP files.** It uses **pre-built route files (.chn/.chf)** loaded at map start. Routes are manually edited in-game or auto-linked from path_corner entities.

**Key takeaway for qbots:** 3ZB2's route graph structure (node-based with adjacency lists) is excellent, but you must build the graph from BSP geometry, not from in-game editing.

---

## Architecture Overview

### Data Structures

**Route Node** (`bot.h:307-318`):
```c
#define MAXNODES 10000
#define MAXLINKPOD 6

typedef struct {
    vec3_t Pt;                    // Target point (x,y,z)
    union {
        vec3_t Tcourner;          // For trains/grapple
        unsigned short linkpod[MAXLINKPOD]; // Connected node indices
    };
    edict_t *ent;                 // Associated entity (door, train, item)
    short index;                  // Node index
    short state;                  // GRS_NORMAL, GRS_ITEMS, etc.
} route_t;

extern route_t Route[MAXNODES];
int CurrentIndex; // Number of active nodes
```

**Node States:**
- `GRS_NORMAL` - Standard walkable area
- `GRS_ITEMS` - Item location
- `GRS_TELEPORT` - Teleporter destination
- `GRS_PUSHBUTTON` - Button to press
- `GRS_ONPLAT` - Elevator/platform
- `GRS_ONTRAIN` - Moving platform/train
- `GRS_GRAPSHOT` - Grappling hook point
- `GRS_REDFLAG` / `GRS_BLUEFLAG` - CTF flag locations

## Route Loading

### Binary Format (`g_ctf.c:LoadChainFile 3020-3169`)

**Files:**
- DM maps: `3zb2/chdtm/<mapname>.chn`
- CTF maps: `3zb2/chctf/<mapname>.chf`

**Format:**
```c
// Header
char signature[8] = "3ZBRGDTM";  // Magic number
int CurrentIndex;                 // Number of nodes

// Route array
for(i=0; i<CurrentIndex; i++) {
    fwrite(&Route[i].Pt, sizeof(vec3_t), 1, f);
    fwrite(&Route[i].linkpod, sizeof(short), MAXLINKPOD, f);
    fwrite(&Route[i].state, sizeof(short), 1, f);
    // Entity pointers re-associated after load by coordinate matching
}
```

**Loading Process:**
1. Read header and verify signature
2. Load `CurrentIndex` and route array
3. Re-associate entity pointers by matching coordinates with world entities
4. Initialize state machines for special nodes (elevators, trains, etc.)

## Route Building

### Manual Editing (In-Game)

**Commands:**
- `chedit 1` - Enable route editing mode
- Touch `path_corner` entities to add nodes
- Touch items to add GRS_ITEMS nodes
- Touch teleporters to add GRS_TELEPORT nodes

**Entity Spawning:**
```c
// path_corner (g_misc.c:501)

    // Becomes route point when touched in edit mode
    // Added to Route[] array with GRS_NORMAL state
}

// Items (g_items.c:1080)
void Touch_Item(edict_t *ent, edict_t *other) {
    if (other->client->chedit) {
        // Add item location as GRS_ITEMS route point
        Route[CurrentIndex].Pt = ent->s.origin;
        Route[CurrentIndex].state = GRS_ITEMS;
        Route[CurrentIndex].ent = ent;
        CurrentIndex++;
    }
}
```

### Auto-Linking Algorithm (`g_spawn.c:G_FindRouteLink 797-899`)

**Process:**
```c
// After spawning, auto-link nearby nodes
for(i=0; i<CurrentIndex; i++) {
    if(Route[i].state != GRS_NORMAL) continue;
    
    for(j=0; j<CurrentIndex; j++) {
        if(abs(i-j) <= 50) continue;  // Don't link too-close nodes
        
        VectorSubtract(Route[j].Pt, Route[i].Pt, v);
        
        // Distance/height filters
        if(VectorLength(v) > 200) continue;           // Max horizontal
        if(v[2] > JumpMax || v[2] < -500) continue;   // Vertical tolerance
        
        // Check if jump is physically possible
        if(needs_jump && !RTJump_Chk(i, j)) continue;
        
        // Check if already linked to nearby node
        if(AlreadyLinkedNearby(i, j)) continue;
        
        // Line-of-sight check
        trace = gi.trace(Route[j].Pt, NULL, NULL, Route[i].Pt, ent, MASK_SOLID);
        if(trace.fraction == 1.0) {
            // Add bidirectional link
            AddLinkToPod(i, j);
            AddLinkToPod(j, i);
        }
    }
}
```

**Key Parameters:**
- `JumpMax ≈ 400` units (from `VEL_BOT_JUMP=340` and gravity)
- Max horizontal distance: 200 units
- Vertical tolerance: -500 to +400
- `MAXLINKPOD = 6` links per node

### Jump Validation (`g_spawn.c:RTJump_Chk 739`)

```c
qboolean RTJump_Chk(int from, int to) {
    vec3_t v;
    float time, height;
    
    VectorSubtract(Route[to].Pt, Route[from].Pt, v);
    height = v[2];
    
    // Simulate jump trajectory
    time = VectorLength(v) / VEL_BOT_JUMP;
    height_check = (VEL_BOT_JUMP * time) - (0.5 * GRAVITY * time * time);
    
    return (height_check >= height);  // Can the jump clear the gap?
}
```

## Movement & Tracing

### Line-of-Sight (`bot/bot.c:Bot_trace 57`)

```c
qboolean Bot_trace(edict_t *ent, edict_t *other) {
    vec3_t start, end;
    trace_t tr;
    
    VectorCopy(ent->s.origin, start);
    start[2] += ent->viewheight;
    
    VectorCopy(other->s.origin, end);
    end[2] += other->viewheight;
    
    // Special handling for tall entities
    if(other->maxs[2] >= 32)
        end[2] = other->mins[2] + 24;
    
    tr = gi.trace(start, NULL, NULL, end, ent, 
                  CONTENTS_SOLID | CONTENTS_WINDOW | 
                  CONTENTS_LAVA | CONTENTS_SLIME);
    
    return (tr.fraction == 1.0);
}
```

### Navigation Execution

**Path Following:**
1. Bot maintains `routeindex` (current node)
2. Move toward `Route[routeindex].Pt`
3. When reached, check `Route[routeindex].linkpod[]` for next node
4. Select next node based on goal (item, enemy, flag, etc.)
5. Special state handling for elevators, trains, etc.

**State-Specific Logic:**
```c
switch(Route[routeindex].state) {
    case GRS_ONPLAT:
        // Wait for platform to move, then ride
        if(platform->moveinfo.state != STATE_TOP)
            return;  // Wait
        // Ride platform to destination
        break;
        
    case GRS_TELEPORT:
        // Enter teleporter trigger
        Use_Teleporter(ent, Route[routeindex].ent);
        break;
        
    case GRS_ITEMS:
        // Collect item
        Touch_Item(Route[routeindex].ent, ent);
        break;
}
```

## Multi-Level Map Handling

### Stairs/Ramps
- Represented as sequences of closely-spaced route nodes
- Each step ~16-32 units apart
- Auto-linking connects them if within distance/height limits

### Elevators/Platforms
- `GRS_ONPLAT` state with entity pointer
- Bot waits for platform to reach destination
- Rides platform by staying on it
- Disembarks when platform reaches top/bottom

### Trains
- `GRS_ONTRAIN` state with entity pointer
- Similar to elevators but for moving platforms
- Uses `trainteam` chaining (G_FindTrainTeam in g_spawn.c:599)

### Jumps
- `RTJump_Chk` validates vertical transitions
- If jump is too high/impossible, no link is created
- Bots must find alternative route (stairs, elevator)

### Teleporters
- `GRS_TELEPORT` state
- Bot walks into trigger, teleport happens automatically
- Appears at destination teleporter

## Critical Differences from qbots Requirements

### 3ZB2 (Gamecode DLL)
- ✅ Uses `gi.trace()` for collision detection
- ✅ Accesses `g_edicts[]` for entity positions
- ✅ Reads entity state directly (platform position, door state)
- ✅ Route editing in-game
- ❌ **Cannot be used by qbots** (requires server internals)

### qbots (External Client)
- ❌ No `trace()` access - must use BSP geometry
- ❌ No entity access - only receives PVS-limited updates
- ✅ Must parse `.bsp` file directly for world model
- ✅ Must build route graph from BSP brushes/leafs
- ✅ LOS calculations must use BSP tree traversal
- ❌ Cannot edit routes in-game - must pre-compute from BSP

## What qbots Can Reuse

1. **Route graph structure** - Node-based with adjacency lists (`linkpod[]`)
2. **Linking algorithm** - Distance/height filters + LOS trace is effective
3. **Jump validation** - Physics-based `RTJump_Chk` prevents impossible jumps
4. **State machine** - Different node types need special handling
5. **Pre-computation** - Routes should be built offline, not at runtime

### What qbots Must Build Differently

1. **Node generation** - Parse BSP brushes/leafs instead of in-game editing
2. **LOS checks** - Use BSP tree traversal instead of `gi.trace()`
3. **Entity access** - Build entity table from server frames instead of `g_edicts[]`
4. **Platform detection** - Parse BSP special brushes or observe entity traffic
5. **Route files** - Build from BSP analysis, not from in-game editing

## Binary Format Example

**aerowalk.chn (first 100 bytes):**
```
Offset 0:  "3ZBRGDTM"     (8 bytes) - Magic signature
Offset 8:  0x000001F4     (4 bytes) - CurrentIndex = 500 nodes
Offset 12: Route[0].Pt    (12 bytes) - vec3_t position
Offset 24: Route[0].linkpod (12 bytes) - 6 x short (0-5 linked nodes)
Offset 36: Route[0].state (2 bytes) - Node type
Offset 38: Padding        (58 bytes) - Reserved zeros
Offset 96: Route[1].Pt    (12 bytes) - Next node
...
```

Each node: **96 bytes** total (12 + 12 + 2 + 58 + 12 padding)
